# Research: Go Conversion for Podimo-to-RSS

## Research Question
Convert the Podimo-to-RSS Python web service into a Go application that compiles to a single static binary. Preserve all core functionality: HTTP Basic Auth, Podimo GraphQL client, RSS feed generation, disk caching, rate limiting, and anti-bot HTTP resilience.

## Summary

A Go rewrite eliminates Python's interpreter/dependency overhead and produces a single deployable binary. The translation is straightforward for most components, with three areas requiring deliberate decisions:

1. **HTTP client / anti-bot** вҖ” No native Go `cloudscraper` exists. The best approach is `azuretls-client` (or `tls-client`) for TLS/browser fingerprint mimicry, plus optional ZenRows/ScraperAPI proxy, plus an optional FlareSolverr sidecar for JS challenges.
2. **RSS generation** вҖ” `github.com/eduncan911/podcast` natively supports all iTunes tags including `<itunes:image>` at channel and item level, eliminating the Python `lxml` post-processing workaround entirely.
3. **Caching** вҖ” Go has no direct `diskcache` equivalent. A lightweight file-based KV store with TTL (JSON or gob files, or `bbolt`/`badger`) replaces the SQLite-backed cache.

The remaining components (HTTP server, GraphQL client, configuration, logging, rate limiting, cookie jars) all have simple Go standard-library or minimal-dependency equivalents.

## Current Python Architecture (Baseline)

### HTTP Routes (`main.py`)
- `/health` вҖ” JSON health probe
- `/search?q=` вҖ” Podcast search via GraphQL `podcastsAutocomplete` with 3 fallback query variants
- `/subscriptions` вҖ” User's followed podcasts via `podcastsFollowed`
- `/` вҖ” Form UI (GET/POST) returning feed URL
- `/feed/<podcast_id>.xml` вҖ” Primary feed endpoint with Basic Auth or `LOCAL_CREDENTIALS`
- `/feed/<username>/<password>/<podcast_id>.xml` вҖ” Alternate path-embedded credential endpoint
- 404 handler вҖ” Plaintext usage example

### Auth Flow (`podimo/client.py:85-148`)
3 sequential GraphQL POSTs:
1. `AuthorizationPreregisterUser` вҶ’ `preauth_token`
2. `OnboardingQuery` вҶ’ `prereg_id`
3. `AuthorizationAuthorize` вҶ’ `token` (valid ~5 days)

### RSS Generation (`main.py:371-448`)
- `feedgen.feed.FeedGenerator` with `podcast` extension
- Episode processing in chunks of 5 with `asyncio.gather`
- Per-episode HEAD request for `Content-Length`/`Content-Type` (3 retries, 1s backoff)
- `lxml.etree` post-processing to inject `<itunes:image>` elements that `feedgen` rejects

### Caching (`podimo/cache.py`)
- `TOKENS` вҖ” `diskcache.Cache` (or in-memory dict), key=`sha256(username~password)`, TTL=5 days
- `podcast_cache` вҖ” `diskcache.Cache`, key=podcast_id, TTL=6 hours
- `head_cache` вҖ” `diskcache.Cache`, key=episode_id, TTL=7 days
- `cookie_jars` вҖ” in-memory `Dict[str, aiohttp.CookieJar]`
- Lazy TTL eviction on read

### Configuration (`podimo/config.py`)
- `.env` + `os.environ` merge (env overrides `.env`)
- Custom boolean parsing: `bool(str(value).lower() in ['true', '1', 't', 'y', 'yes'])`
- Block list file parsing with `#` comment support

### Error Handling
- Custom exceptions: `PodimoError` вҶ’ `PodcastNotFoundError` / `AuthenticationError`
- Route-level: `PodcastNotFoundError` вҶ’ 404, `PodimoError` вҶ’ 500, generic `Exception` вҶ’ 500

## Go Ecosystem Findings

### 1. HTTP Server Framework

**Recommendation: Standard library `net/http`**

The current Python app uses Quart (async Flask-like) with Hypercorn ASGI server. For Go, the standard library `net/http` with `http.Server` is sufficient and idiomatic. No framework (Gin, Echo, Fiber) is needed because:
- Only 7 routes, all simple
- No middleware chains beyond logging and rate limiting
- Single binary goal favors minimal dependencies

If we want routing helpers, `gorilla/mux` or `go-chi/chi` are lightweight options, but even `http.ServeMux` in Go 1.22+ supports path parameters and method matching natively.

### 2. RSS / Podcast Feed Generation

**Recommendation: `github.com/eduncan911/podcast` (v1.4.x)**

| Library | iTunes Support | Maintenance | Verdict |
|---------|---------------|-------------|---------|
| `eduncan911/podcast` | Full (channel + item images, author, block, duration, category) | Stable, maintenance mode | **Use** |
| `gorilla/feeds` | None | Active | Not suitable вҖ” zero iTunes namespace |
| `jbub/podcasts` | Partial (unclear item images) | Stale (2021) | Avoid |

`eduncan911/podcast` natively emits:
- `<enclosure>` with typed constants (`podcast.MP3`, etc.)
- `<itunes:image href="...">` at both channel and item level
- `<itunes:author>`, `<itunes:block>`, `<itunes:duration>`
- `<itunes:category>` with subcategories
- Automatic `xmlns:itunes` namespace injection

This **eliminates the need for XML post-processing** that the Python app does with `lxml`. The `ValueError` catch-and-inject workaround for `.webp`/extensionless image URLs becomes unnecessary.

**XML post-processing fallback** (if needed): `github.com/beevik/etree` вҖ” pure-Go, Python-ElementTree-like API, handles namespaces correctly.

**Key libraries:**
- `azuretls-client` вҖ” Session-based, Chrome-by-default, HTTP/2&3, proxy support, automatic cookie jar
- `tls-client` (bogdanfinn) вҖ” `net/http`-like interface, JA3/JA4 fingerprinting, cookie jar
- `go-retryablehttp` (hashicorp) вҖ” Automatic retries with exponential backoff
- `net/http/cookiejar` вҖ” Standard cookie jar (in-memory)

**Cookie persistence across restarts:** Neither standard library nor `azuretls-client` persists cookies to disk. We'll need a small wrapper (~50 lines) to serialize jar contents to JSON on shutdown and restore on startup, or use a file-backed store.

### 4. GraphQL Client

**Recommendation: Raw `net/http` + `encoding/json` with a ~40-line wrapper**

The Python app uses raw POST with JSON body `{"query": "...", "variables": {...}}` and parses into generic dicts. No schema-first client is needed.

Go equivalent:
```go
type graphqlClient struct {
    endpoint string
    httpClient *http.Client
}

func (c *graphqlClient) Query(ctx context.Context, query string, vars map[string]any, resp any) error {
    // POST {"query": query, "variables": vars}
    // Check status, unmarshal {"data": ..., "errors": ...}
    // Return first error from GraphQL errors array
}
```

Response parsing: Use `map[string]any` for fully dynamic access, or small per-query structs with `json.RawMessage` for deferred parsing of varying shapes.

**Avoid:**
- `machinebox/graphql` вҖ” effectively unmaintained, imports archived `pkg/errors`
- `shurcooL/graphql` вҖ” struct-first, requires defining Go structs for every query shape, fights the ad-hoc string pattern
- Codegen tools (`genqlient`, `gqlgenc`) вҖ” overkill for ~6 ad-hoc queries

### 5. Caching

**Recommendation: Simple file-based KV with TTL using JSON or gob serialization**

Go has no direct `diskcache` equivalent (SQLite-backed KV with TTL). Options:

| Approach | Pros | Cons |
|----------|------|------|
| **JSON/gob files per key** | Zero dependencies, simple, human-readable (JSON) | Slower for very high write volume |
| `bbolt` (BoltDB) | Fast, transactional, embedded | Extra dependency, B-tree not needed |
| `badger` (dgraph) | LSM-tree, very fast | Heavy dependency, overkill |
| `bigcache` / `ristretto` | In-memory, fast | No persistence |

For this workload (low write volume, small values, TTL on read), **file-based JSON is sufficient and keeps dependencies minimal**.

Implementation sketch:
```go
type FileCache struct {
    dir string
}

type cacheEntry struct {
    ExpiresAt time.Time `json:"expires_at"`
    Value     any       `json:"value"`
}

func (c *FileCache) Get(key string) (any, bool) { /* read file, check TTL */ }
func (c *FileCache) Set(key string, value any, ttl time.Duration) { /* write file */ }
```

### 6. Configuration

**Recommendation: `github.com/spf13/viper`**

Viper supports:
- `.env` file loading
- Environment variable override (automatic, with prefix)
- Type casting (bool, int, string)
- Default values

Or, for minimal dependencies, manual `os.Getenv` + a small `.env` parser (there are tiny zero-dep libraries, or we can use `godotenv` which is the Go equivalent of Python's `python-dotenv`).

**Custom boolean parsing** from Python must be preserved:
```go
func parseBool(v string) bool {
    switch strings.ToLower(v) {
    case "true", "1", "t", "y", "yes":
        return true
    default:
        return false
    }
}
```

### 7. Logging

**Recommendation: `log/slog` (Go 1.21+ standard library)**

Structured logging with JSON or text output, level support, and attribute groups. No external dependency needed.

**Recommendation: In-memory map with `sync.RWMutex`**

Direct translation of the Python pattern:

### 9. HTML Templates

**Recommendation: Standard library `html/template`**

The Python app uses Jinja2 (`index.html`, `feed_location.html`). Go's `html/template` is a direct equivalent with automatic HTML escaping for security.

### 10. Testing

Go's standard `testing` package with `net/http/httptest` for HTTP handler tests. For the GraphQL client, use `httptest.Server` to mock Podimo's API responses.

### Data flow
1. Request вҶ’ route handler
2. Basic auth extraction (or LOCAL_CREDENTIALS)
3. Region/locale validation against hardcoded lists
4. Block list check
5. Initialize PodimoClient (load token from cache, load/create cookie jar)
6. If no token вҶ’ 3-step GraphQL auth (sequential)
7. If token вҶ’ proceed to fetch
8. Fetch podcast with pagination (serial loop, limit=100)
9. Cache podcast result
10. For each episode chunk (size=5):
    - Extract audio URL
    - HEAD request for content-length/type (with retry)
    - Add to RSS feed
11. Encode RSS to XML
12. Return `text/xml`

## Performance Considerations

### Go advantages over current Python
1. **No GIL** вҖ” True parallelism for concurrent feed requests
2. **No async/sync bridge** вҖ” `net/http` is natively sync; goroutines handle concurrency. No `run_in_executor` overhead.
3. **Static binary** вҖ” Single file deployment, no virtualenv/pip issues
4. **Memory efficiency** вҖ” Lower baseline memory than Python + Quart + Hypercorn

### Risks
1. **Cloudflare challenges** вҖ” If Podimo serves JS challenges, Go cannot solve them in-process. Must rely on proxy services or FlareSolverr sidecar. The Python `cloudscraper` advantage is lost.
2. **Cookie jar non-persistence** вҖ” Go's `cookiejar` is in-memory. Container restarts lose session state, forcing re-auth. Need custom persistence wrapper.
3. **RSS image alignment** вҖ” Python had a race condition in concurrent image URL injection. Go's goroutine model with explicit channels/sync makes this easier to get right.
