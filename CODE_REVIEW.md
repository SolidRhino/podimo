# Podimo to RSS — Code Review

> Reviewed: 2025  
> Scope: All Python source, templates, config, Docker, CI/CD, docs

---

## Executive Summary

**Podimo to RSS** is a small, single-purpose Python web service that reverse-engineers the Podimo mobile GraphQL API to expose exclusive podcasts as standard RSS feeds. It is a clever utility that solves a real user problem, but the codebase carries several **actual bugs**, **security concerns**, and **maintainability issues** that would bite you in production.

It is clearly a hobby/tinker project (originally by Thijs Raymakers, now forked by `izu-x`) rather than hardened infrastructure. It works, but it needs a cleanup pass before I'd trust it to handle real user credentials at scale.

---

## Architecture & Design (6/10)

The structure is admirably simple for what it does:

- **Entry:** `main.py` — Quart web server, routes, RSS generation
- **API Client:** `podimo/client.py` — GraphQL login & episode fetching
- **Config:** `podimo/config.py` — env var / `.env` loading
- **Cache:** `podimo/cache.py` — disk-backed token, podcast, and HEAD caches
- **Utils:** `podimo/utils.py` — helpers (header generation, async wrapping)

**Good:**
- Clean separation of concerns.
- Multi-layer caching (tokens, podcasts, HEAD metadata) is well thought out.
- GraphQL pagination is implemented (`limit`/`offset` loop).

**Bad:**
- The async/sync split is messy. `cloudscraper` and `ZenRowsClient` are synchronous libraries shoehorned into async via `run_in_executor`. This is functional but wasteful — a new `cloudscraper` session is created on every single feed request.
- There are **two competing authentication schemes** in the URL design:
  1. `/feed/<podcast_id>.xml` with HTTP Basic Auth (generated by the UI)
  2. `/feed/<username>/<password>/<podcast_id>.xml` (defined but unused by the UI)

  The second leaks credentials into server access logs (they appear in the URL path).

## Real Bugs Found

### 🐛 Critical: `return ValueError(...)` instead of `raise`
**File:** `podimo/client.py:39`
```python
if not is_correct_email_address(username):
    return ValueError("Email is not in the correct format")  # BUG
```
This **returns** a `ValueError` *object* instead of raising it. The caller receives a `ValueError` instance assigned to their `client` variable and continues execution, which will explode later with confusing attribute errors.

**Fix:**
```python
raise ValueError("Email is not in the correct format")
```

---

### 🐛 Critical: Fragile `getPodcastName`
**File:** `podimo/client.py:159`
```python
def getPodcastName(self, podcast):
    return list(podcast.values())[1]["title"]
```
This assumes the GraphQL response dict always has at least 2 values and that the second insertion order is reliable. Dictionary ordering is guaranteed in modern Python, but relying on `list(dict.values())[1]` is extremely brittle. If Podimo adds a new top-level field, this breaks.

**Fix:** Access by known key:
```python
def getPodcastName(self, podcast):
    return podcast.get("podcast", {}).get("title", "Unknown")
```

---

### 🐛 Critical: Content-Type Logic Is Backwards
**File:** `main.py:206-211`
```python
content_type, _ = guess_type(url)
if content_type is None and 'content-type' in response.headers:
    content_type = response.headers['content-type']
else:
    content_type = 'audio/mpeg'
```
If `guess_type()` successfully infers a type from the URL (e.g. `.mp3` → `audio/mpeg`), the `else` branch **overwrites it** with `'audio/mpeg'`. That's harmless for MP3s, but if the URL was an `.m4a` or `.ogg`, `guess_type` would return the correct type and this code would *ignore* it, forcing `audio/mpeg`.

**Fix:**
```python
content_type, _ = guess_type(url)
if content_type is None:
    content_type = response.headers.get('content-type', 'audio/mpeg')
```

---

### 🐛 Moderate: Empty Episode List = Broken RSS
**File:** `main.py:259-281`
If `len(episodes) > 0` is false, **none of the feed metadata is set** (no title, description, image, language, etc.). `feedgen` will emit a malformed/empty RSS feed.

**Fix:** Move the `if len(episodes) > 0` block or handle the empty case gracefully.

---

### 🐛 Moderate: ZenRowsClient Instantiated Per Request
**File:** `podimo/client.py:68-69`
```python
elif ZENROWS_API is not None:
    scraper = ZenRowsClient(ZENROWS_API)
```
This creates a new client object on **every GraphQL post**, which is wasteful and slow.

**Fix:** Create once at module level or store it on `PodimoClient` initialization.

---

### 🐛 Moderate: `cache` Parameter Collision
**File:** `main.py:2` and `podimo/cache.py`
`main.py` does `import podimo.cache as cache`, but the module is imported as a namespace. The `cache.py` module also defines a global named `cache` (the `diskcache.Cache` instance). This is confusing but not immediately broken.

**Fix:** Rename the import or the module-level variable for clarity.

---

### 🐛 Moderate: Regex Pattern Too Permissive
**File:** `main.py:96`
```python
podcast_id_pattern = re.compile(r"[0-9a-fA-F\-]+")
```
This matches `"---"`, `"abc"`, or `""` (well, empty string fails `fullmatch`). The template uses a strict UUID regex, but this server-side validation is too loose.

**Fix:** Use a strict UUID pattern matching the template:
```python
podcast_id_pattern = re.compile(r"[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}", re.I)
```

## Security Concerns (5/10)

| Issue | Severity | Notes |
|-------|----------|-------|
| **Credentials in URL** | High | In non-local mode, the web UI generates a URL like `https://user:pass@host/...`. This is a known pattern for private RSS feeds, but it means passwords live in browser history, bookmarks, and referrer headers. At minimum, the UI should flash a loud warning. |
| **No HTTPS enforcement** | Medium | The tutorial literally tells users to point `PODIMO_HOSTNAME` at their raw public IP. Combined with cleartext HTTP Basic Auth, credentials are sent unencrypted over the internet. |
| **CORS wildcard** | Low | `Access-Control-Allow-Origin: *` on every response, including RSS feeds. Unnecessary for a self-hosted tool. |
| **No rate limiting** | Medium | Anyone can hammer the `/feed/...` endpoint. Because each request triggers Podimo GraphQL calls, this is abusable. |
| **Synchronous scraper per request** | Low | Each feed request spawns a `cloudscraper` session. If exploited, this consumes threads and memory quickly. |
| **Block list is substring match** | Low | `if any(item in request.url for item in BLOCKED):` — clever but could false-positive if a UUID happens to contain another blocked string. |

## Code Quality & Maintainability (5/10)

**The Good:**
- Environment-based configuration is solid (`.env` + env vars).
- `tutorial.md` is a nice inclusion for non-technical users.
- Dark mode CSS in templates is a pleasant surprise.
- Docker multi-arch build (amd64, arm64, armv7, armv6) shows care.

**The Rough:**
- String-matching on exception messages (`if "Podcast not found" in exception:`) instead of checking structured error codes.
- `split_username_region_locale` silently defaults to Dutch (`'nl'`, `'nl-NL'`) if the username doesn't contain exactly 2 commas. Arbitrary magic fallback.
- `@app.after_request` logs "Incoming request" — after the request has already been processed.
- `werkzeug~=2.3.7` is pinned, likely because Quart 0.18.x has compatibility issues. This debt will grow.
- The `Makefile` generates a systemd service file inline with `cat > .podimo.service`. It works, but it's unusual.
- `.env.example` has `DEBUG=true` uncommented by default — not great for a production template.

## Overall Verdict

**It works, but don't expose it to the internet as-is.**

This is a classic "tinkerer's project" — functional, clever (the GraphQL reverse-engineering is neat), and useful, but with rough edges that become real problems under pressure:

1. **Fix the `return ValueError` bug immediately.** It's embarrassing and will cause confusing failures.
2. **Harden `getPodcastName`.** Access response fields by key name, not dictionary insertion order.
3. **Switch to a single aiohttp-based client** instead of synchronous `cloudscraper` + `async_wrap`. Or at least reuse the scraper session across requests.
4. **Add input validation** to `serve_basic_auth_feed` when `LOCAL_CREDENTIALS=True` — currently missing region/locale are passed through unchecked until `serve_feed` rejects them.
5. **Clean the HEAD cache mime-type logic.** It's backwards.
6. **Add a warning** in the UI when credentials will be embedded in the URL.

If you're the maintainer, this is a solid 2-hour refactor away from being trustworthy. If you're a user, run it behind a reverse proxy (Caddy/nginx) with HTTPS, use `LOCAL_CREDENTIALS=true`, and don't expose it to the public internet.
